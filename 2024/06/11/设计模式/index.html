菜鸟设计模式

1.创造型模式

1.1.单例模式（Singleton Pattern）

  • 单例模式：是一种创造型设计模式，它确保一个类只有一个实例，并且提供了一个全局访问点来访问该实例
  • 特点：
    1.单例类只能有一个实例
    2.单例类必须自己创建自己的唯一实例
    3.单例类必须给所有其他对象提供这一个实例
  • 缺点：不能继承，与单一职责原则冲突，一个类应该只关心内部逻辑，而不应该关心实例化方式
  • 类图

1.1.1.单例模式的几种实现方法

  1. 懒汉式、线程不安全：get时才创建实例，没有考虑多线程
  2. 懒汉式、线程安全：get时才创建实例，通过加锁考虑多线程
  3. 饿汉式：直接在类装载的时候就创建实例
  4. 双重校验锁（DCL，double checked locking）：安全，并且在多线程下保持高性能
    为什么要有双重校验：因为某一个线程可能在第一次检查和第二次检查之间开始锁，其他线程就停止在锁之前，解锁后，如果不判断，就会产生多个实例，而不是单例模式了。

1.2.工厂方法模式（Factory Parrten）

  • 工厂方法模式：提供了一种创建对象的方法，将对象的创建和对象的使用分离开来
  • 特点：
    将对象的创建逻辑封装在一个工厂类里面，而不是在客户端代码里面直接实例化对象，这样提高了代码的可维护性和可拓展性
  • 缺点：每增加一个产品时，需要增加一个具体类和对应的工厂，使得类成倍增多增加系统复杂度和类的依赖

1.4.建造者模式（Builder Pattern）

  • 建造者模式：把一个复杂对象的构建过程与其表示相分离，从而可以创建具有不同表示形式的对象。
  • 特点：将对象构建过程和表示分离，并且以一定顺序给构建对象赋予一定功能特定
  • 结构：
  1. 产品：要构造的复杂对象，通常具有多个功能或者行为
  2. 抽象建造者：定义构建对象的抽象接口，包括构建产品的各个部分的方法
  3. 具体建造者：实现抽象建造者接口，具体确定如何构建产品各个部分，并且返回构建的对象
  4. 指导者：负责调用建造者的方法来构建产品，指导者不了解具体实现，只是关心构建顺序和方法
  • 类图
  • 代码实现：
  • 一些优化
  1. 利用std::unique_ptr和std::make_unique进行智能管理内存
  2. 利用const定义一些常量返回值等等
  3. 析构函数设置为虚函数，使得派生类对象可以正确释放
  • 建造者模式和抽象工厂模式的对比
  1. 建造者模式创建的是具有多种属性或者特定行为的对象，而抽象工厂模式创建的是一类具有单一属性的对象群
  2. 建造者模式在以一定顺序给对象以属性，而抽象工厂模式是具体类有一个特定属性或者行为
  3. 建造者模式通过指导者来控制构建过程，而抽象工厂模式由客户端直接创建具体工厂来创建对象

1.5.原型模式（Prototype Pattern）

  • 原型模式：用于创建重复的对象，也就是对当前已有对象克隆，同时又能保证性能。
  • 特点：可以运行时动态删除或者建立模型，提高了性能，避免了构造函数的约束。通常与工厂模式方法一起使用
  • 缺点：
  1. 配备克隆方法需要全面考虑类的功能，对已有类可能比较难实现，特别是处理不支持串行化的间接对象或具有循环结构的引用时
  2. 必须实现Clone接口
  • 结构：
  1. 原型接口：一个用于克隆自身的接口
  2. 具体原型类：实现原型接口的具体类，负责实际的克隆操作。
  3. 客户端：使用原型实例来创建新的对象，而不是通过构造函数

2.结构型模式

2.1.适配器模式（Adapter Pattern）

  • 适配器模式：通过一个中间件适配器将一个类的接口转换成用户所需要的接口，使原本不能一同工作的类可以协同工作
  • 特点：可以将两个不兼容的类协同合作，提高类的复用性
  • 缺点：可能导致系统结构混乱，难以维护和理解
  • 结构：
  1. 目标接口：客户所要使用的类
  2. 适配者类：一个已经存在的接口，它不能满足目前要求，需要适配
  3. 适配器类：通过组合或者继承的方式，实现目标接口

2.2.装饰器模式（Decorator Pattern）

  • 允许向一个现有的对象添加功能，同时又不改变其现有的结构
  • 特点：允许运行时动态添加或者修改对象的功能
  • 缺点：多层装饰器可能导致系统复杂性增加
  • 结构：
  1. 抽象组件：定义了原始组件和装饰器对象的公共接口或抽象类，可以是具体组件的父类或接口
  2. 具体组件：是被装饰的原始对象，定义了需要添加新功能的对象
  3. 抽象装饰器：继承自抽象组件，包含一个抽象组件对象，并且定义了与抽象组件相同的接口，同时可以通过组合的方式持有其他装饰器对象
  4. 具体装饰器：实现了抽象装饰器，向抽象组件添加新的功能。一般会在调用原始对象的方法前或者方法后执行自己的操作
  • 类图

  • 一些优化：
  1. 使用智能指针进行自动化内存释放
  2. const表示不修改任何成员变量

2.3.代理模式（Proxy Pattern）

  • 代理模式：为其他对象提供一种代理，并且开放接口以控制对这个对象的访问
  • 特点：将访问控制与业务逻辑分离，可以灵活添加额外的功能或者控制，可以智能处理访问请求，如延迟加载、缓存等
  • 缺点：增加代理层可能会影响请求的处理速度，某些类型的代理模式可能实现起来比较复杂
  • 结构：
  1. 抽象主题：定义了真实主题和代理主题的共同接口，这样在任何使用真实主题的地方都可以使用代理主题
  2. 真实主题：实现了抽象主题接口，是代理对象所代表的真实对象。
  3. 代理：实现了抽象主题接口，并且持有对抽象主题的引用。代理主题实际上就是在真实主题的基础上添加一些功能
  4. 客户端：使用抽象主题接口来实现访问真实主题或者代理主题，不知道具体是哪一个实现类

2.4.外观模式（Facade Pattern）

  • 外观模式：为一个复杂的子系统提供一个一致的高层接口。
  • 特点：客户端可以通过简化的接口与子系统进行交互，而不需要知道子系统的内部细节与复杂性；降低了客户端与子系统之间的耦合度
  • 缺点：违反了开闭原则，对子系统的修改可能会对外观类进行一些更改
  • 结构
  1. 外观：提供一个简化接口，封装了子系统的复杂性。客户端直接与接口进行交互，而不是和复杂的子系统
  2. 子系统：由多个相互关联的类组成，负责系统的具体功能。外观对象通过调用这些子系统来完成客户端请求
  3. 客户端：使用外观对象来与子系统进行交互，而不需要了解子系统内部的具体实现

2.5.桥接模式（Bridge Pattern）

  • 桥接模式：将抽象部分与实现部分分离，使得可以分别独立变化
  • 特点：提供更灵活的扩展方式，避免使用继承导致的类爆炸；提高系统灵活性和扩展性；
  • 缺点：增加了系统的理解与设计难度；要求开发者在抽象层进行设计与编程
  • 结构：
  1. 抽象：定义抽象接口，通常包含对实现接口的引用
  2. 扩展抽象：对抽象的扩展，可以是抽象类的子类或者具体实现类，里面有具体实现类的指针以使用
  3. 实现：定义实现接口，提供基本操作的接口
  4. 具体实现：实现实现接口的具体类

2.6.组合模式（Composite Pattern）

  • 组合模式：又叫部分整体模式，是用于把一组相似的对象当作一个单一的对象。
  • 特点：
  1. 将对象组合成树形结构以表示“部分-整体”的层次结构。
  2. 客户端可以统一处理所有类型的节点
  3. 可以轻松添加新的叶子类型或树枝类型
  • 缺点：违反依赖倒置原则，组件的声明是基于具体类而不是接口，可能导致代码灵活性较低
  • 结构：
  1. 组件：定义了组合中所有对象的通用接口，可以是抽象类或者接口。它声明了访问和管理子组件的方法，如添加删除获取等。
  2. 叶子节点：表示组合中的叶子节点对象，叶子节点没有子节点。它实现了组件接口的方法，但通常不包含子组件
  3. 复合节点：表示组合中的复合对象，复合节点可以包括子节点，也可以是叶子节点，也可以是其他复合节点。它实现了组件接口的方法，包括管理子组件的方法。
  4. 客户端：通过组件接口和组合结构进行交互，客户端不需要区分叶子节点和复合节点，可以一直对待整体和部分

2.7.享元模式（Flyweight Pattern）

  • 享元模式：主要用于减少创建对象的数量，以减少内存占用和提高性能。
  • 特点：
  1. 通过共享对象来减少创建大量相似对象时的内存消耗
  2. 对象的状态可以外部化，即对象的部分状态可以独立于对象本身存在
  3. 减少对象创建时间，提高系统效率
  • 缺点：
  1. 需要分离内外部状态，增加了设计和实现的复杂性
  2. 可能产生线程安全问题
  • 结构：
  1. 享元工厂：负责创建和管理享元对象，通常包含一个池（缓存）用于存储和复用已经创建的享元对象
  2. 抽象享元：定义了具体享元和非共享享元的接口，通常包含了设置外部状态的方法
  3. 具体享元：实现了抽象享元接口，包含内外部状态。内部状态是可以共享的，外部状态是由客户端传递的
  4. 客户端：通过享元工厂获取享元对象，并通过设置外部享元状态来操作享元对象。客户端通常不需要关心享元对象的具体实现

3.行为型模式

3.1.策略模式（Strategy Pattern）

  • 策略模式：定义了一系列算法，将每一个算法分别封装到独立的类中，使得他们可以相互替换。
  • 特点：将一系列相似算法封装，避免大量if…else；这一些列的类之区别仅仅在于他们的行为
  • 缺点：
  1. 每增加一个算法，就要增加一个策略类
  2. 所有策略类都必须要对外暴露，以方便被选择和利用
  • 结构：
  1. 环境：维护对一个策略对象的引用，负责将客户端请求委派给具体的策略类对象执行。环境类可以通过依赖注入、简单工厂方式获取具体策略对象
  2. 抽象策略类：定义了策略对象的公共接口或抽象类，规定了具体策略类必须实现的方法
  3. 具体策略类：实现了抽象策略类定义的接口或者抽象类，包含了具体的方法

3.2.观察者模式（Observer Pattern）

  • 观察者模式：定义了一种一对多的依赖关系，当一个对象的状态发生改变时，其所有依赖者都会收到通知并且自动更新。
  • 特点：无需对每一个对象进行更新，而是有变化就自动更新；观察者和主题之间是抽象耦合的
  • 缺点：
  1. 性能问题：如果观察者过多，通知过程会耗时
  2. 循环依赖：可能导致循环调用和系统崩坏（可以考虑使用异步执行）
  3. 缺乏变化详情：观察者不知道主题如何变化，只知道变化发生
  • 结构：
  1. 主题：也被称为被观察者或可观察者，它是具有状态的对象，并且维护着一个观察者列表，主题提供了添加、删除和通知观察者的方法
  2. 观察者：观察者是接收主题通知的对象，观察者有一个更新的方法，当收到主题的通知时，需要调用该方法进行更新操作
  3. 具体主题：具体实现类，维护观察者列表，并且在状态更新时通知观察者
  4. 具体观察者：实现更新方法，定义了收到主题状态更新时要执行的具体操作

3.3.命令模式（Command Pattern）

  • 命令模式：将请求封装为一个对象，允许用户使用不同的请求对客户端进行参数化
  • 特点：
  1. 请求者和执行者之间的耦合度降低
  2. 新命令可以很容易添加到系统中
  3. 允许用户使用不同请求对客户端进行参数化
  • 缺点：系统可能产生过多具体命令类，增加系统复杂度
  • 结构：
  1. 命令：定义了执行操作的接口，通常包含一个execute方法，用于调用具体的操作
  2. 具体命令：实现了命令接口，负责执行具体的操作。通常包含了对接收者的引用，通过调用接收者的方法来完成请求的处理
  3. 接收者：知道如何执行与请求相关的操作，实际执行命令的对象
  4. 调用者/请求者：发送命令的对象，它包含了一个命令对象并能触发命令的执行。调用者并不直接处理请求，而是将通过将请求传递给命令对象来实现
  5. 客户端：创建具体命令对象并设置其接收者，将命令对象交给调用者执行

3.4.责任链模式（Chain of Responsibility Pattern)

  • 责任链模式：为请求创建了一个接收者对象的链。
  • 特点：
  1. 允许将请求沿着处理者链传递，直到请求被处理为止
  2. 降低耦合度，对接收者和发送者进行解耦
  3. 对象不需要直到链结构
  4. 可以通过改变链的成员或者顺序，动态地新增或者删除责任，易于扩展
  • 缺点：
  1. 不能保证请求一定被链中某一个处理者接收
  2. 可能影响系统性能，且调试困难，有可能导致循环调用
  3. 运行时特征不明显，可能妨碍除错
  • 结构：
  1. 抽象处理者：定义一个处理请求的接口，通常包含一个处理请求的方法和下一个处理者的引用
  2. 具体处理者：实现了抽象处理者接口，负责处理请求。如果能够处理该请求，直接处理；否则，将请求传递给下一个处理者
  3. 客户端：创建处理者对象，并将他们连接成一条责任链。通常客户端仅需将请求发送给责任链的第一个处理者，不需要关心请求的处理过程。

3.5.模板模式（Template Pattern）

  • 模板模式：一个抽象类公开定义了执行它的方法的方式/模板。它的子类可以按需重写方法，但是调用以抽象类中定义的方式进行
  • 特点：
  1. 父类定义了算法骨架，允许子类在不改变算法结构的前提下重定义算法的某些特定步骤
  2. 子类可以扩展或者修改算法的可变部分
  • 缺点：每一个不同的实现都需要一个类，可能导致系统复杂
  • 结构：
  1. 抽象父类：定义了一些模板方法或抽象方法或具体方法
  2. 具体子类：继承自抽象父类，并且实现抽象方法
  3. 钩子方法：在抽象父类中定义，可以被子类重写，以影响模板方法的行为
  4. 客户端：使用抽象父类和具体子类，无需关心模板方法的细节

3.6.状态模式（State Pattern）

  • 状态模式：类的行为是基于它的状态改变的
  • 特点：
    1、允许一个对象在其内部状态发生改变的时候改变其行为，看起来就像是改变了其类一样
  1. 将状态转换逻辑封装在状态对象内部，避免用大量条件语句来切换行为
  2. 将所有与特定状态相关的行为集中到一个类中
  3. 允许多个上下文共享同一个状态
  • 缺点：
  1. 每个状态都需要单独创建一个类
  2. 模式结构和实现相对复杂
  3. 增加新状态或者修改状态行为可能需要修改现有代码
  • 结构：
  1. 上下文：定义了客户端感兴趣的接口，并且维护一个当前状态对象的引用。上下文可以通过状态对象来委托处理状态相关的行为
  2. 状态：定义了一个接口，用于封装与上下文相关的一个状态行为
  3. 具体状态：实现了状态接口，负责处理与该状态相关的行为。具体状态对象通常会在内部维护一个对上下文对象的引用，以便根据不同条件切换到不同状态

3.7.访问者模式（Visitor Pattern）

  • 访问者模式：使用访问者类，改变元素类的执行算法
  • 特点：
  1. 将数据结构与在该数据结构上面的操作进行分离，从而使得添加新的操作变得容易，而不需要修改数据结构本身
  2. 每一个访问者负责一项职责
  3. 访问者可以独立于数据结构进行变化
  • 缺点：
  1. 元素需要向访问者公开信息
  2. 元素类需要维持与访问者的兼容
  3. 访问者模式依赖具体类而不是接口
  • 结构：
  1. 抽象访问者：定义了访问元素的接口
  2. 具体访问者：实现访问者接口，提供对每个具体元素类的访问和相应操作
  3. 抽象元素：定义了一个接受访问者的方法
  4. 具体元素：实现元素接口，提供接受方法，允许访问者访问并且操作
  5. 对象结构（可选）：定义了如何组装元素，如一个组合类
  6. 客户端：使用访问者模式对对象结构进行操作

3.8.中介者模式（Mediator Pattern）

  • 中介者模式：定义了一个中介对象来封装一系列对象之间的交互，用于降低多个对象和类之间的通信复杂性
  • 特点：
  1. 引入了中介对象来封装和协调多个对象之间的交互,从而降低对象间的耦合度
  2. 将一对多关系简化成了一对一关系
  3. 对象只知道中介者对象，不知道其他对象
  • 缺点：
  1. 中介者可能变得庞大复杂，难以维护
  2. 中介者一旦出现问题，整个系统就容易瘫痪
  • 结构：
  1. 抽象中介者：定义了一个接口，用于与各个同事对象通信，并且管理各个同时对象之间的关系。通常包括一个或者多个事件处理方法，用于处理各种交互事件
  2. 具体中介者：实现了中介者接口，负责实现各个同时对象之间的通信逻辑。他会维护一个对各个同时对象的引用，并协调他们的交互
  3. 抽象同事对象：定义了一个接口，用于与中介者通信。通常包含一个发送信息的方法，以及一个接收信息的方法。
  4. 具体同事对象：实现了同事对象接口，是真正参与到交互中的对象，它会将消息发送给中介者，然后中介者转发给其他同事对象

3.9.迭代器模式

  • 迭代器模式：顺序访问一个聚合对象里的每一个元素，但又不用暴露其内部的表示
  • 特点：
  1. 不同迭代器可以迭代不同的遍历方式
  2. 聚合类不需要关系遍历逻辑
  3. 可以对同一个聚合对象进行多次遍历
  4. 增加新的聚合类和迭代器类都很方便，无需更改现有代码
  • 缺点：每增加一个聚合类，就需要增加至少一个迭代器类，增加系统复杂性
  • 结构：
  1. 迭代器接口：定义了访问和遍历每一个聚合类的方法，通常包括获取下一个元素、判断是否还有元素、获取当前位置的方法
  2. 具体迭代器：实现了迭代器接口，负责对聚合对象进行遍历和访问，同时记录遍历的当前位置
  3. 聚合对象接口：定义了创建迭代器对象的接口，通常包含一个工厂方法，用于创建迭代器对象
  4. 具体聚合对象：实现了聚合对象接口，负责创建具体的迭代器对象，并提供需要进行遍历的数据

3.10.备忘录模式

  • 备忘录模式：保存一个对象的某个状态，以便在需要时恢复其状态
  • 特点：
  1. 在不破坏封装性的前提下，捕获一个对象的内部状态，并且允许在对象之外保存和恢复这个状态
  2. 允许用户方便地回到历史状态
  3. 用户不需要关心状态的保存细节
  • 缺点：如果保存对象的状态复杂，保存状态可能占用较多资源
  • 结构：
  1. 备忘录：负责存储原发器对象的内部状态，备忘录可以保持原发器的状态的一部分或者全部信息
  2. 原发器：创建一个备忘录对象，并且可以使用备忘录对象存储或者恢复自身的状态。原发器通常会在需要保存状态的时候才创建备忘录对象，在需要恢复的时候使用备忘录对象
  3. 负责人：负责保存备忘录对象，但不对备忘录对象进行操作或者检查，负责人只负责把备忘录对象传递给其他对象
